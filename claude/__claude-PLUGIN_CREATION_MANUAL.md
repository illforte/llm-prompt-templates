# AI Plugin Creation Manual

**A reusable prompt for transforming any existing skill/command into a marketplace-ready plugin — deployable on Claude, Gemini, and ChatGPT.**

Copy this entire document into a new Cowork session alongside your existing skill files. Claude will follow it step-by-step.

---

## Cross-Platform Architecture Overview

> Read this section before starting. It defines the strategy the phases implement.

### The Three Platforms

| Platform | Instructions Layer | Tool Layer | Marketplace |
|---|---|---|---|
| **Claude** | `SKILL.md` + `plugin.json` | MCP server (stdio) | Cowork plugin store |
| **Gemini ADK** | `agent.yaml` (YAML) | MCP server or OpenAPI tools | Workspace Studio (org-internal) |
| **Gemini Gems** | Plain text instructions (UI only) | None (prompt-only) | Peer-to-peer share link |
| **ChatGPT Custom GPT** | Plain text instructions (UI only) | GPT Actions (OpenAPI 3.x) | GPT Store |
| **ChatGPT MCP App** | Prompt + MCP config | MCP server (same as Claude) | ChatGPT app store (2026) |

### The Key Insight: MCP Is the Universal Tool Standard

All three major AI platforms have adopted the **Model Context Protocol (MCP)** as of 2025-2026:
- **Claude**: Native (stdio MCP in `settings.json`)
- **Gemini ADK**: MCP supported in agent config
- **OpenAI**: MCP Apps launched Oct 2025; ChatGPT app store opened Dec 2025

**This means**: If you build your tools as an MCP server (Phase 5), that server runs on all three platforms without modification. **The tool layer is free.** The only porting work is adapting the instruction text.

### What "Cross-Platform" Actually Means

```
Your plugin
│
├── mcp/server.py          ← Works as-is on Claude, Gemini ADK, ChatGPT MCP Apps
│
├── skills/your-plugin/
│   └── SKILL.md           ← Claude format (this manual)
│
├── platform-adapters/     ← Generated by `make gemini` / `make chatgpt` (Phase 6.5)
│   ├── gemini/
│   │   ├── agent.yaml     ← Gemini ADK (YAML manifest)
│   │   └── gem.md         ← Gemini Gems (plain text export)
│   └── chatgpt/
│       ├── instructions.md  ← Custom GPT system prompt
│       ├── openapi.yaml     ← GPT Actions spec (generated from MCP tools)
│       └── mcp-config.json  ← ChatGPT MCP App config
```

### Platform Comparison: What's Different

| Concern | Claude | Gemini ADK | ChatGPT |
|---|---|---|---|
| **Instruction format** | Structured SKILL.md | YAML `agent.yaml` | Plain text (UI) |
| **Tool registration** | `settings.json` or `plugin.json` mcpServers | `agent.yaml` tools section | GPT Actions (OpenAPI) or MCP |
| **Credential storage** | File + env var + keychain | Env var + ADC + service account | OAuth 2.0 or API key (per action) |
| **Marketplace** | Cowork plugin store | Workspace Studio (internal) | GPT Store (requires verification) |
| **Data persistence** | Local files via `_get_data_dir()` | Cloud (Vertex AI) or local | No persistence in Custom GPTs |
| **Sandbox network** | May be blocked (see Phase 4.5) | Cloud-hosted, no sandbox block | Cloud-hosted, no sandbox block |

### Scope Decision

Before Phase 0, decide your deployment targets:

- **Claude only**: Follow Phases 0–8, skip Phase 6.5 and Phase 9
- **Claude + Gemini**: Follow all phases; Phase 6.5 generates adapters, Phase 9 Section A has the spec details
- **Claude + ChatGPT**: Follow all phases; Phase 6.5 generates adapters, Phase 9 Section B has the spec details
- **All three**: Follow all phases, full Phase 6.5 + Phase 9

## Sprint Orchestration

> This section turns the manual from a linear prompt (which LLMs skim toward the end) into sprint-sized work packages that each get full attention.

### Why Sprints

A 2000-line prompt exceeds most LLMs' max-effort window. The result: early phases get deep coverage, later phases get shortcuts. Sprints fix this by:

1. **Bounding scope** — each sprint is 2–4 phases, completable in one session at max effort
2. **Forcing depth** — the sprint gate blocks progression until all criteria are verified
3. **Enabling parallelism** — independent phases within a sprint run as sub-agents concurrently
4. **Clean handoff** — if a session hits its effort ceiling, state is saved for the next session

### Sprint Plan

Execute sprints in order. Each sprint MUST pass its gate before the next sprint begins.

| Sprint | Phases | Sub-Agent Parallelism | Focus |
|---|---|---|---|
| **Sprint 1: Foundation** | 0, 1, 2 | Phase 0 first → then Phases 1+2 in parallel | Discovery, structure, config isolation |
| **Sprint 2: Skill Layer** | 3, 4, 4.5 | Sequential (each builds on previous) | Shell compat, SKILL.md, Cowork conventions |
| **Sprint 3: Tool Layer** | 5, 5.5 | Sequential (5 then 5.5 hardens it) | MCP server + security hardening |
| **Sprint 4: Distribution** | 6, 6.5 | Phase 6 first → then 6.5 uses its outputs | Installer + cross-platform build |
| **Sprint 5: Ship It** | 7, 8, 9 | Phase 7 first → then 8 launches 3 parallel audit agents → then 9 | Repo hygiene, deep audit, cross-platform export |

### Sprint Execution Protocol

At the start of **each sprint**:

```
1. PLAN    — Read all phases in this sprint. List every subtask with estimated effort.
2. EXECUTE — Work through phases. Launch sub-agents for parallelizable work.
3. GATE    — Verify every success criterion for every phase in the sprint.
4. LOG     — Record: [SPRINT N GATE: ✓ | list of criteria verified]
5. HANDOFF — If session limit reached: save state (see template below).
```

### Sub-Agent Orchestration

Use sub-agents (Claude `Agent` tool, Gemini parallel function calls, ChatGPT multi-tool) for independent work within a sprint:

**Sprint 1 example:**
```
Agent 1: Phase 0 — Discovery & Inventory
  → wait for results →
Agent 2: Phase 1 — Plugin Directory Structure  (parallel)
Agent 3: Phase 2 — Config Isolation            (parallel)
  → wait for both → Sprint 1 Gate
```

**Sprint 5 example (Phase 8 deep audit):**
```
Agent 1: Security Audit     (parallel)
Agent 2: Architecture Audit  (parallel)
Agent 3: Compatibility Audit (parallel)
  → wait for all 3 → merge findings → fix → re-audit → Sprint 5 Gate
```

**Rules:**
- Only parallelize phases that have NO data dependency between them
- Each sub-agent gets the FULL phase instructions (not a summary)
- Sub-agent results are verified against the phase's success criteria before merging
- If a sub-agent fails, retry it — don't skip and move on

### Sprint Gate (Mandatory)

After all phases in a sprint are complete, run this gate. **The next sprint CANNOT begin until the gate passes.**

```markdown
## Sprint N Gate

### Phase X.X: [name]
- [ ] Criterion 1 from phase success criteria — [VERIFIED: evidence]
- [ ] Criterion 2 — [VERIFIED: evidence]
...

### Phase Y.Y: [name]
- [ ] Criterion 1 — [VERIFIED: evidence]
...

### Gate Result
- All criteria verified: YES / NO
- Blockers: [list any failures]
- Confidence: [XX.X%]
- Proceed to Sprint N+1: YES / NO
```

### Anti-Shortcut Rules

These rules exist because LLMs predictably cut corners on long workflows:

1. **No skipping phases** — even if a phase "seems unnecessary," run it. Mark findings as "N/A" if truly not applicable.
2. **No combining sprints** — even if the work seems small, complete one sprint gate before starting the next.
3. **No summary-instead-of-doing** — "This phase would involve..." is not completion. Create the actual files.
4. **No confidence inflation** — 99.9% means you re-read every file and verified every criterion. If you didn't, score lower and iterate.
5. **Sub-agents must return evidence** — "looks good" is not evidence. Return file paths, command outputs, or diff snippets.

### Session Handoff Template

If a session hits its effort ceiling mid-sprint, save this state file at `.claude/sprint-state.json` (or equivalent):

```json
{
  "manual": "AI Plugin Creation Manual",
  "version": "1.7",
  "current_sprint": 3,
  "completed_sprints": [1, 2],
  "current_sprint_progress": {
    "phase_5_mcp_server": "completed",
    "phase_5.5_security": "in_progress — input validation done, output sanitization pending"
  },
  "blockers": [],
  "files_created": ["mcp/server.py", "scripts/config_loader.py"],
  "files_modified": ["plugin.json", ".gitignore"],
  "next_action": "Complete Phase 5.5 output sanitization, then run Sprint 3 Gate",
  "timestamp": "2026-04-08T14:30:00Z"
}
```

**Next session**: Load this file first. Resume from `next_action`. Do NOT restart from Sprint 1.

### Confidence Loop Protocol

After completing **each phase** (not just each sprint), perform a self-audit:

1. Re-read every file you created or modified in this phase
2. Verify against the success criteria listed for that phase
3. Assign a confidence score (0-100%)
4. If < 99.9%: identify gaps, fix them, re-score. Repeat until ≥ 99.9%
5. Log your confidence in the phase summary: `[CONFIDENCE: XX.X% → XX.X% → 99.9%+ ✓]`

Minimum 3 rounds per phase. No exceptions.


### Key Principles

- **Skills go in `skills/plugin-name/`** — not the root. The `plugin.json` references them.
- **MCP servers go in `mcp/`** — with their own `requirements.txt`.
- **Scripts go in `scripts/`** — all Python logic, utilities, config loading.
- **References go in `references/`** — templates, static assets, architecture docs.
- **Internal docs are gitignored** — setup guides, implementation notes with company data stay local.

### Critical: SKILL.md Root Symlink

Claude Code discovers skills by looking for `SKILL.md` at `<skill-root>/SKILL.md`. Since your actual SKILL.md lives in `skills/plugin-name/SKILL.md`, you must create a symlink at the plugin root:

```zsh
# From inside the plugin directory:
ln -s skills/plugin-name/SKILL.md SKILL.md
```

This symlink IS committed to the repo (it's safe — it only contains a relative path). The installer script should create it automatically (see Phase 6).

**Without this symlink, the skill will not appear in Claude Code's skill list.**

```json
{

  "version": "0.1.0",
  "author": {
    "name": "your-github-username",
    "email": "your-email@example.com"
  },
  "license": "MIT",
  "homepage": "https://github.com/username/plugin-name",
  "repository": "https://github.com/username/plugin-name",

```

**Variables**: `${CLAUDE_PLUGIN_ROOT}` = plugin install directory. `${CLAUDE_PLUGIN_DATA}` = persistent user data directory (outside the plugin tree).

### Multi-Skill Plugins

Some plugins contain multiple sub-skills (e.g., a pipeline with distinct phases). In this case:

```
plugin-name/
├── SKILL.md                    # Symlink to the MAIN orchestrator skill
├── skills/
│   ├── main-skill/             # Orchestrator (referenced by root SKILL.md)
│   │   └── SKILL.md
│   ├── sub-skill-a/            # Phase/module skill
│   │   └── SKILL.md
│   └── sub-skill-b/
│       └── SKILL.md
```

The root `SKILL.md` symlink should point to the **main orchestrator** skill, which then references sub-skills in its workflow. The sub-skills are loaded by Claude when the orchestrator's instructions reference them — they don't need individual root symlinks.

In `plugin.json`, list all skills:

```json
"skills": [
    "./skills/main-skill",
    "./skills/sub-skill-a",
    "./skills/sub-skill-b"
]
```

### Workspace Integration Pattern

Plugins are typically stored in a development repo (`~/dev-repos/plugin-name/`) and linked into one or more Claude Code workspaces:

```zsh
# Link plugin into a workspace
ln -s ~/dev-repos/plugin-name ~/my-workspace/.claude/skills/plugin-name
```

This symlink means:
- The dev repo stays in `~/dev-repos/` and can be committed to git
- The workspace sees the plugin in `.claude/skills/plugin-name/`
- Runtime data must NOT go into the dev repo (covered in Phase 2)

## Phase 2: Config Isolation (Multi-User / Multi-Company)

**Goal**: Externalize ALL sensitive and user-specific data so the plugin works for anyone.

### The Data Directory Pattern

All runtime data (configs, API keys, logs, caches, reports) must live OUTSIDE the plugin tree in a persistent "data directory". This is critical for:
- Multi-company use (different configs for different orgs)
- Security (secrets never in the repo)
- Clean updates (pull new plugin version without overwriting user data)

### Implementation: `_get_data_dir()`

Add this to your config loader (or create `scripts/config_loader.py`):

```python
import os

def _get_data_dir():
    """
    Resolve persistent data directory. 3-tier fallback:
    1. PLUGINNAME_DATA_DIR env var (explicit override)
    2. CLAUDE_PLUGIN_DATA env var (set by Claude plugin system)
    3. ~/.config/plugin-name/ (XDG-style default)
    """
    # Tier 1: Explicit env var (for multi-company setups)
    data_dir = os.environ.get("PLUGINNAME_DATA_DIR")
    if data_dir:
        data_dir = os.path.expanduser(data_dir)
        os.makedirs(data_dir, exist_ok=True)
        return os.path.abspath(data_dir)

    # Tier 2: Claude plugin system
    plugin_data = os.environ.get("CLAUDE_PLUGIN_DATA")
    if plugin_data:
        plugin_data = os.path.expanduser(plugin_data)
        os.makedirs(plugin_data, exist_ok=True)
        return os.path.abspath(plugin_data)

    # Tier 3: XDG default
    xdg_config = os.environ.get("XDG_CONFIG_HOME", os.path.expanduser("~/.config"))
    data_dir = os.path.join(xdg_config, "plugin-name")
    os.makedirs(data_dir, exist_ok=True)
    return os.path.abspath(data_dir)
```

### Path Placeholders

In config files, use placeholders instead of absolute paths:

| Placeholder | Resolves To | Use For |
|---|---|---|
| `{data_dir}` | `_get_data_dir()` | Logs, caches, API keys, user config |
| `{workspace}` | User's workspace folder | Reading user files |
| `{skill_dir}` | Plugin installation dir | Reading bundled templates/references |

### Config File Search Order

```
1. ENV: PLUGINNAME_CONFIG → explicit path
2. {data_dir}/config.json → user's persistent config
3. {workspace}/config.json → workspace-local config
4. {skill_dir}/scripts/config.template.json → defaults (READ-ONLY)
```

### Scrubbing Checklist

Search and replace ALL of the following in committed code:

- [ ] Company domain names → `example.com`
- [ ] Internal URLs → `your-instance.example.com`
- [ ] Email addresses → `user@example.com`
- [ ] Slack workspace IDs → empty string or placeholder
- [ ] Slack channel IDs → empty string or placeholder
- [ ] User/colleague IDs → empty arrays `[]`
- [ ] Internal ticket patterns → generic examples
- [ ] API keys / tokens → never committed, loaded from data_dir

### Verification

```bash
# Scan for any remaining sensitive patterns
grep -rn "your-company\|internal-url\|slack-workspace-id" --include="*.py" --include="*.json" --include="*.md" --include="*.sh" .
# Must return ZERO matches in committed files
```

### Success Criteria

- [ ] `_get_data_dir()` implemented with 3-tier fallback
- [ ] All paths in config use `{data_dir}` placeholder
- [ ] Config template committed (with placeholders, no real data)
- [ ] Actual config gitignored
- [ ] API key locations point to `{data_dir}` first
- [ ] Zero company-specific strings in committed code (verified by grep)

## Phase 3: Shell Compatibility (macOS/ZSH)

**Goal**: All shell scripts and generated commands must be zsh-safe.

### Why This Matters

macOS Catalina+ defaults to zsh. Bash-only constructs cause cryptic failures:
- `!` in double quotes → `zsh: event not found` (history expansion)
- `${!var}` → syntax error (bash indirect expansion)
- `BASH_SOURCE` → undefined variable
- `read -a` → invalid option

### Fixes

| Bash-only | ZSH-compatible |
|---|---|
| `#!/bin/bash` | `#!/usr/bin/env zsh` |
| `${BASH_SOURCE[0]}` | `$0` |
| `${!ARRAY[@]}` | iterate with `for x in "${ARRAY[@]}"` |
| `[ ! -z "$var" ]` | `[ -n "$var" ]` |
| `echo "Hello!"` | `echo 'Hello!'` or `echo "Hello\!"` |
| `read -a arr` | `read -A arr` |
| `declare -A` | `typeset -A` |

### Guardrail: `scripts/shell_compat.py`

Add a shell compatibility checker to your plugin. It scans commands for zsh pitfalls:

```python
# Usage in Python code:
from shell_compat import check_zsh_safe
issues = check_zsh_safe('echo "Hello!"')  # Returns list of issues

# Usage from CLI:
python3 scripts/shell_compat.py scan install_plugin.sh
python3 scripts/shell_compat.py check 'some command'
```

Add this rule to your SKILL.md under "Critical Rules":

> **ZSH guardrail (macOS)**: All shell commands must be zsh-safe. Never use `!` inside double quotes. Never use bash-only constructs (`${!var}`, `BASH_SOURCE`, `read -a`). Validate with `python3 scripts/shell_compat.py check 'command'` before presenting shell snippets.

### Success Criteria

- [ ] All `.sh` files use `#!/usr/bin/env zsh` shebang
- [ ] Zero bash-only constructs in shell scripts
- [ ] `shell_compat.py` passes scan on all `.sh` files
- [ ] SKILL.md includes zsh guardrail rule

## Phase 4.5: Cowork Sandbox Compatibility

**Goal**: Ensure the plugin works inside Cowork (Claude Desktop sandbox) sessions, not just local CLI.

### The Cowork Sandbox Environment

Cowork sessions run in a sandboxed container with critical constraints:

| Constraint | Detail |
|---|---|
| **Network egress blocked** | `WebFetch` to external services (Redmine, Jira, REST APIs) may be blocked by org policy |
| **Isolated home directory** | `~` resolves to the *sandbox* home, NOT `/Users/username` — tilde-based file checks fail |
| **Host MCP access** | MCP servers run on the **host machine** and communicate via stdio — they bypass network restrictions entirely |
| **Shell env vars** | Shell env vars exported in `~/.zshrc` are NOT automatically inherited by the sandbox but ARE inherited by MCP server subprocesses (which run on the host) |

### Rule: MCP Over WebFetch/curl

Any skill that calls an external API via `curl` or `WebFetch` will fail if the Cowork sandbox has egress restrictions. The correct architecture:

```
Cowork sandbox → mcp__plugin__tool() → Host machine → External API
                 ✓ stdio, always works      ✓ host has network access
```

**Every external API call the skill makes must be wrapped in an MCP tool.** The MCP server runs on the host — no egress restrictions apply.

### Sandbox-Safe Credential Discovery

`~` in the sandbox points to the sandbox's home directory. `Path.home()` in an MCP server works correctly because the MCP server is a host subprocess. Use this discovery order in ALL credential lookups:

```python
import os, subprocess
from pathlib import Path

def get_credential(env_var: str, config_filename: str, keychain_service: str = None) -> str | None:
    """Universal sandbox-safe credential discovery."""

    # 1. Env var FIRST — set in ~/.zshrc, inherited by host MCP subprocess
    key = os.environ.get(env_var, "").strip()
    if key:
        return key

    # 2. Absolute path — works regardless of sandbox ~ resolution
    #    (MCP server runs on host, so Path.home() is correct here)
    candidates = [
        Path(f"/Users/{os.environ.get('USER', 'user')}/.config/{config_filename}"),
        Path.home() / ".config" / config_filename,
        Path(f"/Users/{os.environ.get('USER', 'user')}/.claude/{config_filename}"),
        Path.home() / ".claude" / config_filename,
    ]
    for path in candidates:
        if path.exists():
            key = path.read_text().strip()
            if key:
                return key

    # 3. macOS Keychain
    if keychain_service:
        try:
            result = subprocess.run(
                ["security", "find-generic-password", "-s", keychain_service, "-w"],
                capture_output=True, text=True, timeout=3,
            )
            if result.returncode == 0 and result.stdout.strip():
                return result.stdout.strip()
        except Exception:
            pass

    return None
```

## Phase 5: MCP Server (if applicable)

**Goal**: Wrap your data operations as MCP tools.

### When to Use MCP

Use MCP tools when your plugin needs to:
- Make API calls (Redmine, Jira, Slack, etc.) — **especially important for Cowork sandbox** where WebFetch may be blocked
- Process data that's too complex for inline Python
- Provide structured tool interfaces to Claude
- Maintain state across operations (caches, logs)

### Choosing an MCP Implementation

| Approach | When to Use | Dependencies |
|---|---|---|
| **Stdlib JSON-RPC** | Simple tools, no external deps, Cowork/sandbox priority | None (Python 3.6+) |
| **FastMCP + Pydantic** | Complex tools, rich type validation, async I/O | `mcp>=1.0.0`, `pydantic>=2.0.0`, Python 3.10+ |

**Default to stdlib** unless you specifically need async streaming or complex input validation. The stdlib approach works on any Python version, has zero install requirements, and is simpler to debug.

### Option A: Stdlib JSON-RPC (Recommended for most plugins)

```python
#!/usr/bin/env python3
"""
MCP server for plugin-name — no external dependencies.
Implements MCP stdio protocol (JSON-RPC 2.0) directly.
Runs on HOST machine, bypasses sandbox network restrictions.
"""
import json, os, sys
from pathlib import Path
from typing import Any, Optional

TOOLS = {
    "prefix_action": {
        "description": "What this tool does in one sentence.",
        "inputSchema": {
            "type": "object",
            "properties": {
                "param": {"type": "string", "description": "What this parameter does"},
            },
            "required": ["param"],
        },
        "fn": lambda args: _tool_action(args),
    },
}

def _tool_action(args: dict) -> str:
    param = args.get("param", "")
    # Credential discovery (see Phase 4.5 pattern)
    api_key = os.environ.get("MY_API_KEY") or _read_key_file()
    if not api_key:
        return "Error: No API key found."
    # ... implementation ...
    return json.dumps({"result": "..."}, ensure_ascii=False)

def _read_key_file() -> Optional[str]:
    for path in [Path.home() / ".config" / "my-service" / "api_key"]:
        if path.exists():
            return path.read_text().strip() or None
    return None

def _send(msg: dict):
    sys.stdout.write(json.dumps(msg, ensure_ascii=False) + "\n")
    sys.stdout.flush()

def _handle(req: dict) -> Optional[dict]:
    method, req_id, params = req.get("method",""), req.get("id"), req.get("params",{})
    if method == "initialize":
        return {"jsonrpc":"2.0","id":req_id,"result":{
            "protocolVersion":"2024-11-05","capabilities":{"tools":{}},
            "serverInfo":{"name":"plugin-name","version":"1.0.0"}}}
    if method == "notifications/initialized":
        return None
    if method == "tools/list":
        tools = [{"name":n,"description":s["description"],"inputSchema":s["inputSchema"]}
                 for n,s in TOOLS.items()]
        return {"jsonrpc":"2.0","id":req_id,"result":{"tools":tools}}
    if method == "tools/call":
        name, tool_args = params.get("name"), params.get("arguments",{})
        if name not in TOOLS:
            return {"jsonrpc":"2.0","id":req_id,"error":{"code":-32601,"message":f"Unknown: {name}"}}
        try:
            text = TOOLS[name]["fn"](tool_args)
        except Exception as e:
            text = f"Error: {e}"
        return {"jsonrpc":"2.0","id":req_id,"result":{"content":[{"type":"text","text":text}]}}
    if req_id is not None:
        return {"jsonrpc":"2.0","id":req_id,"error":{"code":-32601,"message":f"Unknown: {method}"}}
    return None

if __name__ == "__main__":
    for line in sys.stdin:
        line = line.strip()
        if not line: continue
        try:
            resp = _handle(json.loads(line))
            if resp: _send(resp)
        except Exception as e:
            _send({"jsonrpc":"2.0","id":None,"error":{"code":-32700,"message":str(e)}})
```

**No `requirements.txt` needed.** Smoke-test with:

```bash

{"jsonrpc":"2.0","id":2,"method":"tools/list","params":{}}' | python3 mcp/server.py
```

```python
from mcp.server.fastmcp import FastMCP
from pydantic import BaseModel, Field

mcp = FastMCP("plugin_name_mcp")

class ToolInput(BaseModel):
    """Input schema — Claude sees these field descriptions."""
    param: str = Field(description="What this parameter does")

@mcp.tool(name="prefix_action", description="What this tool does in one sentence.")
async def tool_action(params: ToolInput) -> str:
    """Implementation."""
    # Use asyncio.to_thread() for blocking I/O
    result = await asyncio.to_thread(blocking_function, params.param)
    return result

if __name__ == "__main__":
    mcp.run(transport="stdio")
```

Use Option B when you specifically need async streaming, rich Pydantic validation, or complex middleware.

### MCP Best Practices

- **Prefix all tools** with a short namespace (e.g., `tga_`, `jira_`, `crm_`)
- **Credential discovery**: Follow the Phase 4.5 pattern — env var first, absolute path second, keychain third
- **Pydantic models for all inputs** (Option B) — Claude sees the field descriptions as tool documentation
- **Return strings** — MCP tools return text that Claude interprets
- **Use `asyncio.to_thread()`** for blocking I/O in Option B (httpx, file reads)
- **Config loading**: Import from `scripts/config_loader.py`, use `_get_data_dir()`
- **Error handling**: Return human-readable error strings, not stack traces
- **Idempotency**: Tools that write data should check before writing (deduplication)
- **Smoke-test before registering**: Run the stdin echo test to verify protocol compliance

### requirements.txt (Option B only)

```
httpx>=0.25.0
```

**Option A (stdlib)**: No `requirements.txt` needed. Works on Python 3.6+, including macOS system Python 3.9.

**Option B**: The `mcp` package requires Python 3.10+. On macOS, the system Python is 3.9 and cannot install this package. See Phase 6 for the multi-strategy installer pattern that handles this.

## Phase 5.5: MCP Security Hardening

**Goal**: Protect your MCP server against the known attack vectors documented in CVE-2025-53109/53110, CVE-2025-68143-68145, CVE-2026-32871, and the OWASP MCP Top 10 (2025).

> MCP servers run with your host user's full privileges. A prompt injection in a Redmine ticket description can weaponize the LLM into calling your tools with attacker-chosen parameters. These hardening steps are not optional.

### 5.5.1 Input Validation (Prevent Path Traversal + Injection)

Every tool parameter that becomes part of a URL path, file path, or shell command must be sanitized:

```python
import re

def _sanitize_path_segment(segment: str) -> str:
    """Strip anything that isn't alphanumeric, dash, underscore, or dot."""
    return re.sub(r'[^a-zA-Z0-9_\-.]', '', str(segment))

def _safe_int(val, default: int, maximum: int) -> int:
    """Parse int with bounds, never crash on bad input."""
    try:
        return min(max(int(val), 1), maximum)
    except (ValueError, TypeError):
        return default
```

**Apply to every tool**:
- IDs (ticket, project, user): `_sanitize_path_segment()` + `.isdigit()` check
- Limits/offsets: `_safe_int()` with upper bound
- Free-text search queries: `urllib.parse.quote()` (never interpolate raw into URLs)
- Project identifiers: allowlist or `_sanitize_path_segment()`

**Reject, don't sanitize-and-continue**: If an ID contains `../` or non-numeric characters, return a clear error rather than silently stripping the bad chars and proceeding.

### 5.5.2 Output Sanitization (Prevent Prompt Injection via Tool Results)

Ticket descriptions, comments, and project names returned by your MCP server are injected into the LLM context. An attacker who can write a Redmine ticket can embed instructions the LLM will follow.

**Mitigations** (defense-in-depth — no single layer is sufficient):

```python
def _sanitize_output(text: str, max_length: int = 10000) -> str:
    """Reduce prompt injection surface in tool output."""
    # 1. Truncate excessively long content
    if len(text) > max_length:
        text = text[:max_length] + f"\n[TRUNCATED at {max_length} chars]"
    # 2. Strip common injection markers (not foolproof, but raises the bar)
    # These patterns appear in known prompt injection payloads
    text = re.sub(r'<\s*/?(?:system|instruction|prompt|role)[^>]*>', '[STRIPPED]', text, flags=re.IGNORECASE)
    return text
```

Apply to all user-generated content before returning:
- Issue descriptions
- Comment bodies
- Custom field values

**Do NOT sanitize**: structured fields (status names, priorities, dates) — these come from Redmine's controlled vocabulary.

### 5.5.3 Read-Only by Default

**Critical rule**: MCP servers should be read-only unless write access is explicitly required. Every write tool (create ticket, post comment, log time) must:

1. Be gated behind a separate "write mode" flag or a distinct MCP server
2. Validate the target project against an allowlist (see the `BLOCKED_PROJECT_PATTERNS` in `joules-redmine-server.py`)
3. Use dry-run by default (return what *would* happen, require explicit confirmation)
4. Audit-log every write operation

The Redmine MCP server we built is **read-only by design** — this is its strongest security property.

### 5.5.4 Credential Isolation

| Practice | Why |
|---|---|
| Never log the API key (even in debug output) | Keys in logs get committed, copied, shared |
| Use file permissions 600 on key files | Prevents other users/processes from reading |
| Prefer keychain over plaintext files | Keychain is encrypted at rest |
| Never pass keys as command-line arguments | Visible in `ps aux` output |
| Block `ANTHROPIC_BASE_URL` override | CVE-2026-21852: redirects all API calls to attacker |

### 5.5.5 URL Construction Safety

The `REDMINE_URL` (or any base URL) must be configurable but validated:

```python
import urllib.parse

REDMINE_URL = os.environ.get("REDMINE_URL", "https://redmine.einsundnull.de").rstrip("/")

# Validate: must be HTTPS, must be a known host
_parsed = urllib.parse.urlparse(REDMINE_URL)
if _parsed.scheme != "https":
    print("WARNING: REDMINE_URL is not HTTPS — credentials will be sent in plaintext", file=sys.stderr)
```

Use `urllib.parse.urlencode()` for query parameters — never string-format user input into URLs.

### 5.5.6 Known CVE Reference

| CVE | Lesson for your plugin |
|---|---|
| CVE-2025-53109/53110 | Always resolve symlinks; never trust string prefix for path containment |
| CVE-2025-68144 | Never pass LLM-generated strings as CLI arguments without sanitizing |
| CVE-2026-32871 | URL-encode path params; don't string-format into API URLs |
| CVE-2026-21852 | Don't let project configs override base URLs of trusted services |
| CVE-2025-6514 | Validate OAuth callback URLs; never exec() URL components |

### Success Criteria

- [ ] Every tool parameter that enters a URL path is sanitized (`_sanitize_path_segment`)
- [ ] Numeric IDs validated with `.isdigit()` — reject non-numeric with clear error
- [ ] Limits/offsets parsed with `_safe_int()` (bounded, no crash on bad input)
- [ ] Query parameters built with `urllib.parse.urlencode()` (no f-string interpolation)
- [ ] User-generated content in output truncated and stripped of injection markers
- [ ] Server is read-only (no write tools) OR write tools are gated + audit-logged
- [ ] API key never logged, never in CLI args, file permissions 600
- [ ] Base URL validated as HTTPS

### Python Version: Critical Detail

The `mcp` Python package requires **Python 3.10+**. macOS ships with Python 3.9 as system Python, which cannot install `mcp`. The installer MUST handle this:

```zsh
PY_VERSION=$(python3 --version 2>&1 | cut -d' ' -f2)

    warn "System Python $PY_VERSION — need 3.10+ for 'mcp' package"

```

**Multi-strategy dependency installation** (handles system Python 3.9):

```zsh
VENV_DIR="$SCRIPT_DIR/.venv"
PYTHON_CMD="python3"

install_deps() {
    local method=$1
    case $method in
        "venv")
            python3 -m venv "$VENV_DIR" 2>/dev/null || return 1
            source "$VENV_DIR/bin/activate"
            pip install -q mcp pydantic httpx 2>&1 || { deactivate; return 1; }
            PYTHON_CMD="$VENV_DIR/bin/python3"
            deactivate; return 0 ;;
        "user")
            pip3 install --user -q mcp pydantic httpx && return 0 || return 1 ;;
        "brew")
            # Try installed Homebrew Pythons (3.13, 3.12, 3.11)
            for ver in python@3.13 python@3.12 python@3.11; do
                BREW_PY="$(brew --prefix $ver 2>/dev/null)/bin/python3"
                [ -f "$BREW_PY" ] || continue
                "$BREW_PY" -m venv "$VENV_DIR" && \
                source "$VENV_DIR/bin/activate" && \
                pip install -q mcp pydantic httpx && \
                PYTHON_CMD="$VENV_DIR/bin/python3" && \
                deactivate && return 0
                deactivate 2>/dev/null
            done
            return 1 ;;
    esac
}

# Try in order until one works
for strategy in venv user brew; do
    install_deps "$strategy" && break
done
```

```zsh
    if ! grep -qF "$export_line" "$HOME/.zshrc" 2>/dev/null; then
        warn "API key export missing from ~/.zshrc — adding..."
        # (call zshrc append function)
    fi

    echo "Self-healing complete."
}
```

### Idempotent ~/.zshrc Append

```zsh
append_zshrc_if_missing() {
    local marker="$1"  # unique string to grep for
    local line="$2"    # full line to append
    if ! grep -qF "$marker" "$HOME/.zshrc" 2>/dev/null; then
        echo "" >> "$HOME/.zshrc"

        ok "Already in ~/.zshrc: $marker"
    fi
}
```

### Build Script: `scripts/build_platforms.py`

This script queries the MCP server live for its tool schemas, then generates all adapter files. No hand-maintained schema files.

```python
#!/usr/bin/env python3
"""
build_platforms.py — generate Gemini and ChatGPT adapter files from the
Claude plugin structure. Queries mcp/server.py directly for tool schemas.

Usage:
    python3 scripts/build_platforms.py --platform gemini
    python3 scripts/build_platforms.py --platform chatgpt
    python3 scripts/build_platforms.py --platform all
"""

import argparse
import json
import os
import re
import subprocess
import sys
import textwrap
from pathlib import Path

try:
    import yaml  # pip install pyyaml (listed in scripts/requirements-build.txt)
except ImportError:
    yaml = None  # ChatGPT openapi.yaml generation will fall back to json

ROOT = Path(__file__).parent.parent  # plugin root


# ── Load source data ──────────────────────────────────────────────────────────

def load_plugin_meta() -> dict:
    return json.loads((ROOT / ".claude-plugin" / "plugin.json").read_text())


def load_skill_md(plugin_name: str) -> str:
    path = ROOT / "skills" / plugin_name / "SKILL.md"
    if not path.exists():
        path = ROOT / "skills" / plugin_name / "SKILL_SLIM.md"
    return path.read_text()


def strip_skill_frontmatter(skill_md: str) -> str:
    """Remove YAML frontmatter (--- ... ---) from SKILL.md."""
    if skill_md.startswith("---"):
        end = skill_md.find("---", 3)
        if end != -1:
            return skill_md[end + 3:].lstrip("\n")
    return skill_md


def remove_mcp_tool_refs(text: str) -> str:
    """Replace mcp__server__tool references with plain tool names."""
    return re.sub(r'mcp__\w+__(\w+)', r'`\1`', text)


def query_mcp_tools() -> list:
    """
    Start mcp/server.py, send initialize + tools/list, return tool list.
    Works with any compliant MCP server (stdlib or FastMCP).
    """
    server_path = ROOT / "mcp" / "server.py"
    if not server_path.exists():
        return []

    stdin = (
        '{"jsonrpc":"2.0","id":1,"method":"initialize",'
        '"params":{"protocolVersion":"2024-11-05","capabilities":{},'
        '"clientInfo":{"name":"build","version":"1"}}}\n'
        '{"jsonrpc":"2.0","method":"notifications/initialized","params":{}}\n'
        '{"jsonrpc":"2.0","id":2,"method":"tools/list","params":{}}\n'
    )
    try:
        proc = subprocess.run(
            [sys.executable, str(server_path)],
            input=stdin, capture_output=True, text=True, timeout=10,
        )
        for line in proc.stdout.splitlines():
            try:
                resp = json.loads(line)
                if resp.get("id") == 2:
                    return resp["result"]["tools"]
            except (json.JSONDecodeError, KeyError):
                continue
    except Exception as e:
        print(f"Warning: could not query MCP tools: {e}", file=sys.stderr)
    return []


# ── Gemini adapter generators ─────────────────────────────────────────────────

GEMINI_UNSUPPORTED_ATTRS = {"default", "optional", "maximum", "minimum", "oneOf", "anyOf", "$ref"}


def sanitize_schema_for_gemini(schema):
    """Remove OpenAPI attributes Gemini ADK doesn't support. Recurses into dicts and lists."""
    if isinstance(schema, list):
        return [sanitize_schema_for_gemini(item) for item in schema]
    if not isinstance(schema, dict):
        return schema
    return {
        k: sanitize_schema_for_gemini(v)
        for k, v in schema.items()
        if k not in GEMINI_UNSUPPORTED_ATTRS
    }


def tools_to_gemini_function_declarations(tools: list) -> list:
    decls = []
    for tool in tools:
        schema = sanitize_schema_for_gemini(tool.get("inputSchema", {}))
        decls.append({
            "name": tool["name"],
            "description": tool.get("description", "")[:200],  # Gemini ~200 char limit
            "parameters": schema,
        })
    return decls


def build_gemini(meta: dict, skill_text: str, tools: list):
    out_dir = ROOT / "platform-adapters" / "gemini"
    out_dir.mkdir(parents=True, exist_ok=True)

    instructions = remove_mcp_tool_refs(strip_skill_frontmatter(skill_text)).strip()

    # ── agent.yaml ──────────────────────────────────────────────────────────
    mcp_server_rel = "../../mcp/server.py"  # relative from platform-adapters/gemini/
    env_keys = _extract_env_keys(meta)

    env_block = "\n".join(f'        {k}: "${{{k}}}"' for k in env_keys) or "        {}"

    if tools:
        # MCP path (preferred): reference the same server binary
        mcp_block = (
            "tools:\n"
            f"  - mcp:\n"
            f"      command: python3\n"
            f"      args: [\"{mcp_server_rel}\"]\n"
            f"      env:\n{env_block}\n"
        )
    else:
        # No MCP server -- generate inline function_declarations as fallback
        func_decls = tools_to_gemini_function_declarations(tools)
        if func_decls:
            mcp_block = "tools:\n  - function_declarations:\n" + json.dumps(func_decls, indent=4)
        else:
            mcp_block = "# No tools defined"

    agent_yaml = textwrap.dedent(f"""\
        # yaml-language-server: $schema=https://raw.githubusercontent.com/google/adk-python/refs/heads/main/src/google/adk/agents/config_schemas/AgentConfig.json
        name: {meta['name']}
        model: gemini-2.0-flash
        description: >
          {meta.get('description', '')}

        instruction: |
        {textwrap.indent(instructions, '  ')}

        {mcp_block}
        """)
    (out_dir / "agent.yaml").write_text(agent_yaml)
    print(f"  ✓ platform-adapters/gemini/agent.yaml")

    # ── gem.md (Gems UI — prompt only) ──────────────────────────────────────
    gem_md = textwrap.dedent(f"""\
        # {meta['name']} — Gemini Gem

        > Paste this into: gemini.google.com → Gems → Create → Instructions

        ---

        {instructions}

        ---

        > **Note**: This Gem has no tool access. For full API integration, deploy
        > `platform-adapters/gemini/agent.yaml` via Gemini ADK or Workspace Studio.
        """)
    (out_dir / "gem.md").write_text(gem_md)
    print(f"  ✓ platform-adapters/gemini/gem.md")


# ── ChatGPT adapter generators ────────────────────────────────────────────────

def tool_to_openapi_path(tool: dict, base_path: str = "/tools") -> tuple[str, dict]:
    """Convert a single MCP tool definition to an OpenAPI path entry."""
    name = tool["name"]
    description = tool.get("description", "")
    schema = tool.get("inputSchema", {})

    # ChatGPT limits
    summary = description[:295] + "..." if len(description) > 300 else description
    # Sanitize param descriptions to ≤700 chars
    props = schema.get("properties", {})
    for prop in props.values():
        if isinstance(prop, dict) and "description" in prop:
            prop["description"] = prop["description"][:697] + "..." \
                if len(prop["description"]) > 700 else prop["description"]

    path = f"{base_path}/{name}"
    # camelCase operationId
    op_id = re.sub(r'[_-](\w)', lambda m: m.group(1).upper(), name)

    entry = {
        "post": {
            "operationId": op_id,
            "summary": summary,
            "requestBody": {
                "required": bool(schema.get("required")),
                "content": {
                    "application/json": {"schema": schema}
                },
            },
            "responses": {
                "200": {
                    "description": "Tool result",
                    "content": {
                        "application/json": {
                            "schema": {
                                "type": "object",
                                "properties": {
                                    "result": {"type": "string"}
                                },
                            }
                        }
                    },
                }
            },
        }
    }
    return path, entry


def build_chatgpt(meta: dict, skill_text: str, tools: list):
    out_dir = ROOT / "platform-adapters" / "chatgpt"
    out_dir.mkdir(parents=True, exist_ok=True)

    instructions = remove_mcp_tool_refs(strip_skill_frontmatter(skill_text)).strip()
    env_keys = _extract_env_keys(meta)

    # ── instructions.md ─────────────────────────────────────────────────────
    instructions_md = textwrap.dedent(f"""\
        # {meta['name']}

        > Paste into: chatgpt.com → Explore GPTs → Create → Configure → Instructions

        ---

        {instructions}
        """)
    (out_dir / "instructions.md").write_text(instructions_md)
    print(f"  ✓ platform-adapters/chatgpt/instructions.md")

    # ── openapi.yaml (GPT Actions fallback) ─────────────────────────────────
    paths = {}
    for tool in tools:
        path_key, path_val = tool_to_openapi_path(tool)
        paths[path_key] = path_val

    openapi = {
        "openapi": "3.1.0",
        "info": {
            "title": f"{meta['name']} API",
            "description": meta.get("description", ""),
            "version": meta.get("version", "1.0.0"),
        },
        "servers": [{"url": "https://your-api-host.com", "description": "Replace with your API host"}],
        "paths": paths,
    }
    if yaml:
        (out_dir / "openapi.yaml").write_text(yaml.dump(openapi, sort_keys=False, allow_unicode=True))
    else:
        # Fallback: write as JSON with .yaml extension (still valid for GPT Actions import)
        (out_dir / "openapi.yaml").write_text(json.dumps(openapi, indent=2, ensure_ascii=False))
    print(f"  ✓ platform-adapters/chatgpt/openapi.yaml")

    # ── mcp-config.json (ChatGPT MCP App — primary path) ────────────────────
    env_obj = {k: "" for k in env_keys}  # placeholders
    mcp_config = {
        "mcpServers": {
            meta["name"]: {
                "command": "python3",
                "args": [str(ROOT / "mcp" / "server.py")],
                "env": env_obj,
            }
        }
    }
    (out_dir / "mcp-config.json").write_text(json.dumps(mcp_config, indent=2))
    print(f"  ✓ platform-adapters/chatgpt/mcp-config.json")


# ── Helpers ───────────────────────────────────────────────────────────────────

def _extract_env_keys(meta: dict) -> list[str]:
    """Read expected env var names from plugin.json mcpServers env block."""
    keys = []
    for server in meta.get("mcpServers", {}).values():
        keys.extend(server.get("env", {}).keys())
    return keys or ["YOUR_API_KEY"]


# ── Main ──────────────────────────────────────────────────────────────────────

def main():
    parser = argparse.ArgumentParser(description="Generate platform adapter files")
    parser.add_argument("--platform", choices=["gemini", "chatgpt", "all"], required=True)
    args = parser.parse_args()

    meta = load_plugin_meta()
    skill_text = load_skill_md(meta["name"])
    tools = query_mcp_tools()

    print(f"Plugin: {meta['name']} v{meta['version']}")
    print(f"MCP tools found: {len(tools)} ({', '.join(t['name'] for t in tools)})")

    if args.platform in ("gemini", "all"):
        print("Generating Gemini adapters...")
        build_gemini(meta, skill_text, tools)

    if args.platform in ("chatgpt", "all"):
        print("Generating ChatGPT adapters...")
        build_chatgpt(meta, skill_text, tools)

    print("Done.")


if __name__ == "__main__":
    main()
```

## Phase 9: Cross-Platform Export (Gemini + ChatGPT)

**Goal**: Generate Gemini and ChatGPT adapter files from your existing Claude plugin — without duplicating the MCP server.

> **Prerequisite**: Phases 0–8 complete. The MCP server is your tool layer for all platforms. Phase 9 only adapts the instruction and configuration files.

### Section A: Gemini ADK Adapter

Gemini ADK (Agent Development Kit) uses a YAML manifest to define agents. It supports MCP servers directly, so your existing `mcp/server.py` plugs in without changes.

#### A1. Generate `platform-adapters/gemini/agent.yaml`

```yaml
# yaml-language-server: $schema=https://raw.githubusercontent.com/google/adk-python/refs/heads/main/src/google/adk/agents/config_schemas/AgentConfig.json
name: your-plugin-name
model: gemini-2.0-flash
description: >
  [Copy description from plugin.json, adapted to Gemini vocabulary]
instruction: |
  [Convert your SKILL.md to plain instruction text.
   Remove Claude-specific syntax: no YAML frontmatter, no mcp__tool__ references.
   Replace "use mcp__name__tool" with "call the tool 'tool_name'" or describe the action in prose.
   Keep the workflow steps, critical rules, and trigger conditions.]

tools:
  # Option 1: Point directly at your MCP server (recommended)
  - mcp:
      command: python3
      args: ["${AGENT_DIR}/../../mcp/server.py"]
      env:
        YOUR_API_KEY: "${YOUR_API_KEY}"

  # Option 2: Define tools inline using OpenAPI-subset JSON Schema
  # (use only if MCP server isn't available in target env)
  # - function_declarations: [see A2 below]
```

**Variables**: `${AGENT_DIR}` is set by ADK CLI. `${YOUR_API_KEY}` should come from env (ADC, `.gemini/.env`, or service account).

#### A2. Inline Tool Schema (if not using MCP)

If the target Gemini environment doesn't support MCP, generate a `function_declarations` block from your MCP tool definitions:

```yaml
tools:
  - function_declarations:
    - name: prefix_action
      description: "What this tool does. Max ~200 chars for Gemini."
      parameters:
        type: object
        properties:
          param:
            type: string
            description: "What this parameter does"
        required:
          - param
```

**Gemini OpenAPI subset limitations** (do NOT use these — they're unsupported):
- `default` values
- `optional` keyword (use `required` array instead)
- `maximum`/`minimum`
- `oneOf` / `anyOf`
- Nested `$ref`

#### A3. Gemini Gems Export (UI-only, prompt layer only)

Gemini Gems are prompt-only (no tool calling). Generate a plain-text instruction file for users to paste into the Gems UI:

**Generate `platform-adapters/gemini/gem.md`:**

```markdown
# [Plugin Name] — Gemini Gem Instructions

## Persona
[Extract the "who you are" framing from your SKILL.md]

## Task
[Core purpose: what this assistant does]

## Context
[Background, domain knowledge, constraints from your SKILL.md]

## Workflow
[Numbered steps from your SKILL.md Quick Workflow section.
Remove all MCP tool references — describe what the action achieves in natural language.]

## Critical Rules
[Non-negotiable constraints from your SKILL.md]

## Output Format
[How responses should be structured]

---
Note: This Gem does not have tool access. For full tool integration, use the Gemini ADK agent.yaml version.
```

**To deploy**: Go to gemini.google.com → Gems → Create → paste this content as Instructions.

#### A4. Credential Handling for Gemini

| Method | When to Use |
|---|---|
| `export YOUR_API_KEY=...` in shell | Local ADK development |
| `.gemini/.env` file in project | ADK project-level config |
| `gcloud auth application-default login` | Workspace/GCP integration |
| Service account JSON + `GOOGLE_APPLICATION_CREDENTIALS` | CI/CD, production |

The MCP server's credential discovery from Phase 4.5 (env var → absolute path → keychain) works as-is for Gemini ADK, since ADK runs the MCP server as a local subprocess.

### Section B: ChatGPT / OpenAI Adapter

ChatGPT has two integration paths — Custom GPTs (UI-based, consumer) and MCP Apps (developer, emerging standard). Build both.

#### B1. MCP App Config (Recommended — same as Claude)

OpenAI's MCP App format launched Oct 2025 uses the same MCP protocol as Claude. Your existing `mcp/server.py` works without modification.

**Generate `platform-adapters/chatgpt/mcp-config.json`** (for users to reference when connecting):

```json
{
  "mcpServers": {
    "your-plugin-name": {
      "command": "python3",
      "args": ["/absolute/path/to/mcp/server.py"],
      "env": {
        "YOUR_API_KEY": ""
      }
    }
  }
}
```

Users add this to their OpenAI MCP config. The empty env value is a placeholder — users fill in their actual key.

#### B2. GPT Actions OpenAPI Spec (Fallback — for Custom GPTs)

Custom GPTs use GPT Actions (OpenAPI 3.x) instead of MCP. Generate an OpenAPI spec from your MCP tools:

**Generate `platform-adapters/chatgpt/openapi.yaml`:**

```yaml
openapi: "3.1.0"
info:
  title: Your Plugin Name API
  description: >
    [Copy description from plugin.json]
  version: "1.0.0"
servers:
  - url: https://your-api-host.com
    description: Production API
paths:
  /prefix/action:
    post:
      operationId: prefixAction
      summary: What this tool does.          # Max 300 chars
      description: >
        Detailed description for ChatGPT.    # Max 700 chars
      requestBody:
        required: true
        content:
          application/json:
            schema:
              type: object
              properties:
                param:
                  type: string
                  description: What this parameter does
              required:
                - param
      responses:
        "200":
          description: Result text
          content:
            application/json:
              schema:
                type: object
                properties:
                  result:
                    type: string
```

**Key ChatGPT/OpenAPI constraints**:
- Endpoint summary: max 300 characters
- Parameter description: max 700 characters
- `operationId` must be camelCase and globally unique
- No bearer token in spec — auth is configured separately in GPT builder UI

> **Note**: GPT Actions require your API to be publicly accessible via HTTPS. Your MCP server (which runs locally) cannot be used directly as a GPT Action backend — you'd need to deploy the API separately. For most use cases, prefer the MCP App path (B1) which runs locally.

#### B3. Custom GPT Instructions (UI-only)

**Generate `platform-adapters/chatgpt/instructions.md`:**

```markdown
# [Plugin Name]

[Opening sentence: what this GPT does and when to use it.]

## Core Workflow

[Numbered steps from your SKILL.md Quick Workflow.
Replace MCP tool references with natural language descriptions of what the tool does.
Example: "mcp__redmine__get_ticket" → "fetch the ticket details from Redmine"]

## Critical Rules

[Non-negotiable constraints — same as SKILL.md Critical Rules section.
Write as direct imperatives: "Always...", "Never...", "Before any write..."]

## Output Format

[How to format responses: language, structure, length.]
```

**To deploy as Custom GPT**:
1. Go to chatgpt.com → Explore GPTs → Create
2. Switch to **Configure** tab
3. Paste `instructions.md` content into the **Instructions** field
4. Upload any knowledge files from your plugin's `references/` directory
5. Add GPT Actions: paste `openapi.yaml` into the **Actions** section
6. Configure authentication in the Actions panel (API key or OAuth)

#### B4. Credential Handling for ChatGPT

Custom GPTs authenticate via the GPT Actions UI — no file-based credential setup needed. The user connects their key once via OAuth or API key in the GPT builder. The MCP App path inherits the same env var pattern as Claude (set in shell profile).

## Quick Reference: Common Pitfalls

| Pitfall | Symptom | Fix |
|---|---|---|
| `!` in double quotes | `zsh: event not found` | Use single quotes or `\!` |
| `#!/bin/bash` on macOS | Script fails silently or with zsh errors | `#!/usr/bin/env zsh` |
| Hardcoded API URL | Plugin only works for one company | Use `{data_dir}` + config |
| Config in plugin dir | Overwritten on plugin update | Externalize to `_get_data_dir()` |
| GitHub noreply email | `GH007: push declined` | Use `ID+user@users.noreply.github.com` |
| `${!var}` bash expansion | `bad substitution` in zsh | Rewrite loop without indirect |
| Sensitive data in commit | Leaked credentials | Scrub + `.gitignore` + `git filter-branch` |
| Missing `requirements.txt` | MCP server fails to start (Option B) | List all pip dependencies — or switch to stdlib Option A |
| No YAML frontmatter in SKILL.md | Skill not discovered by Claude | Add `---\nname:\ndescription:\n---` |
| No `SKILL.md` at plugin root | Skill invisible to Claude Code | `ln -s skills/plugin-name/SKILL.md SKILL.md` |
| Python 3.9 can't install `mcp` | `No matching distribution found` | Use stdlib Option A (no deps), or multi-strategy install with Homebrew Python |
| MCP only in Claude Desktop | Works in Desktop, not Claude Code | Register in BOTH `claude_desktop_config.json` AND `~/.claude/settings.json` |
| Claude Code entry missing `trust` | Permission prompts on every MCP call | Add `"trust": True` to Claude Code MCP entry |
| No `PLUGINNAME_DATA_DIR` in MCP env | Runtime data saved to dev repo or XDG | Add env var to MCP server registration pointing to workspace data dir |
| Data dir in dev repo | Runtime files get committed | Set `PLUGINNAME_DATA_DIR=<workspace>/.claude/data/<plugin>` and use `_get_data_dir()` |
| Plugin dir is a symlink | `SCRIPT_DIR` resolves to real path, not symlink | Use `cd "$(dirname "$0")" && pwd` — this is correct, workspace detection needs symlink search |
| **WebFetch blocked in Cowork sandbox** | "WebFetch blocked by org policy", session falls back to browser | Wrap ALL external API calls in MCP tools (host process, no egress restriction) |
| **`~` in sandbox resolves to sandbox home** | `~/.config/api_key` check fails → "No API key found" | Check env var FIRST (`os.environ.get`), then absolute paths (`/Users/username/...`) |
| **Skill not visible in Cowork** | Skills only in project `.claude/skills/` missing in Cowork | Also symlink into `~/.claude/skills/plugin-name` (global registration) |
| **API key not inherited by MCP server** | Env var set in terminal but MCP subprocess doesn't see it | Export in `~/.zshrc` (not just `.zshenv`) — `export KEY="$(cat ~/.config/.../key)"` |
| **Skill falls back to browser** | Session says "No API key set up yet, using browser" | Add "Never Fall Back to Browser" rule to SKILL.md; prioritize MCP tools in instructions |
| **Gemini agent.yaml has unsupported OpenAPI** | ADK throws schema error on `default`/`oneOf` | Remove unsupported attrs: default, optional, maximum, oneOf, $ref |
| **`mcp__name__tool` in Gemini/ChatGPT instructions** | AI calls non-existent tool | Adapt instructions per platform — replace with natural language descriptions |
| **GPT Action can't reach local MCP server** | 404 / connection refused | GPT Actions need public HTTPS endpoint; prefer MCP App path (local) for non-public tools |
| **Custom GPT instruction too long** | GPT ignores later rules | Keep under ~2000 tokens; use SKILL_SLIM.md as the source |
| **OpenAPI operationId not unique** | ChatGPT refuses to import schema | Make all operationIds globally unique camelCase per spec |
| **Old ChatGPT Plugin format** | Plugin manifest `.json` 404s | Deprecated April 2024 — migrate to GPT Actions (OpenAPI) or MCP App |
| **Gemini Gems don't call tools** | No API responses | Gems are prompt-only — use ADK agent for tool calling |
| **Path traversal in ticket ID** | `issues/../../admin.json` hits wrong Redmine endpoint | Sanitize all path params: `_sanitize_path_segment()` + `.isdigit()` reject |
| **Prompt injection via ticket description** | LLM follows attacker instructions from Redmine content | Truncate + strip injection markers in output; defense-in-depth |
| **API key in `ps aux` output** | Key visible to other users via process list | Never pass keys as CLI args; use env vars or file reads |
| **MCP server over HTTP (not HTTPS)** | Credentials sent in plaintext | Validate base URL scheme; warn/reject non-HTTPS |
| **Partial install left broken state** | Missing symlinks, MCP not registered, dirs missing | Run `install_plugin.sh --fix` to self-heal; add `--fix` mode to every installer |
| **~/.zshrc export duplicated** | Same export line appears N times after repeated installs | Use `grep -qF` guard before appending (see Phase 6 idempotent pattern) |
